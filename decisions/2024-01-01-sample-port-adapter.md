# 서버 코드에 Port & Adapter 패턴 적용

## 결정
`server/<도메인>/` 폴더를 `model` → `repo`(포트) → `repo.memory`/`repo.supabase`(어댑터) → `service` → `index`(배럴) 레이어로 구성한다.

## 이유
- 비즈니스 로직(`service`)이 저장소 구현(`repo.memory`, `repo.supabase`)에 직접 의존하지 않음
- 테스트 시 메모리 어댑터를 쓰면 DB 없이 빠르게 돌릴 수 있음
- 비개발자가 "서버 코드는 이 폴더 구조를 따르면 된다"고 기계적으로 따라할 수 있음
- 이 패턴이 한번 잡히면 도메인이 늘어나도 일관된 구조 유지

## 고려한 대안
- **단일 파일**: 간단하지만 로직/저장이 섞여서 유지보수 어려움
- **MVC**: 전통적이지만 서비스 레이어 없이 컨트롤러가 비대해지기 쉬움
- **ORM 직접 사용**: Prisma/Drizzle 직접 호출은 간편하지만, 어댑터 교체가 어려움

## 되돌릴 조건
프로젝트 규모가 작아서 레이어 분리가 과하다고 느껴지면 model+service만 남기고 단순화 가능. 하지만 교육 목적상 이 구조를 유지하는 게 좋다.
