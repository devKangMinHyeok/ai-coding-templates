# /ship — 세상에 내보낸다

## 전제
- `/verify`가 초록불이어야 한다. 아니면 먼저 verify부터.
- `/review`가 실행되었어야 한다. 아니면 먼저 review부터.

## 절차

### 1. 자동 리뷰 파이프라인

ship 전에 자동으로 리뷰를 확인한다:

1. **verify 확인**: `pnpm verify`로 타입·린트·테스트 초록불 확인. 실패하면 고치고 반복.
2. **review 확인**: 이번 세션에서 `/review`가 실행되었는지 확인한다.
   - **실행된 적 없음** → `/review`를 자동 실행한다 (변경 파일 기반 프로필 자동 선택).
   - **실행됐지만 Critical이 있었음** → "Critical 항목이 남아있습니다. 수정하셨나요?" 확인.
   - **실행됐고 🟢 판정** → 바로 다음 단계로.
3. **review에서 🔴 판정이면 배포를 멈추고** 사용자에게 알린다.

### 2. 커밋

```bash
git add -A
git commit -m "<무엇을 했는지 한 줄>"
```

커밋 순간 `.husky/pre-commit` 게이트가 자동으로 type-check·lint를 다시 검사한다. **여기서 막히면 통과가 아니다 → 고치고 다시.**

### 3. 푸시

```bash
git push
```

### 4. 배포 확인

**Vercel이 push를 감지해 자동 빌드·배포**한다. 빌드가 깨지면 배포가 실패하므로, 이것도 하나의 안전장치다.

### 5. 결과 안내

배포 URL을 사용자에게 알려준다.

## 자동 기록 (조용히, 확인 안 묻고 기록)

### learnings 자동 기록
배포가 완료되면 이번 작업 세션에서 있었던 주요 발견을 자동 기록한다:
- **에러를 고친 것이 있으면** → 핵심 원인과 해결법을 `bug`로
- **새로운 패턴을 적용한 것이 있으면** → 패턴 요약을 `pattern`으로
- **도구/라이브러리 관련 발견이 있으면** → `tool`로

파일명: `learnings/YYYY-MM-DD-ship-{요약}.md`, frontmatter에 `source: auto/ship`

단, 이미 `/impl` 중에 기록된 내용과 겹치면 중복 기록하지 않는다.
