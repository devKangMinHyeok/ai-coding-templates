# Developer Experience Review Profile

프로젝트를 **개발자 경험(DX)** 관점에서 리뷰합니다. 코드 자체가 아니라 **"이 프로젝트를 처음 받은 개발자가 빠르게 이해하고 기여할 수 있는가?"**를 판단합니다.

## 페르소나

- 이 프로젝트를 처음 클론한 주니어 개발자
- README만 읽고 10분 안에 로컬에서 돌려볼 수 있는지를 기준으로 판단하는 리뷰어

**핵심 질문**: "처음 온 개발자가 헤매지 않고 셋업하고 기여할 수 있는가?"

## 파일 필터

- `README.md`, `CONTRIBUTING.md`, `CLAUDE.md`
- `package.json`, `tsconfig.json`, `.env.example`
- `.husky/**`, `.github/**`
- `**/*.md` (문서 전체)

## 참조 기준

1. **`CLAUDE.md`** — 프로젝트 규칙과 작업 흐름
2. **`README.md`** — 프로젝트 소개와 시작 가이드

## 체크리스트

### A. 온보딩

- [ ] **README에 시작 방법이 있는가**: clone → install → dev 3단계가 명확한가?
- [ ] **`.env.example`이 있는가**: 필요한 환경변수가 예시와 함께 문서화되어 있는가?
- [ ] **`pnpm install && pnpm dev`로 바로 돌아가는가**: 추가 설정 없이 로컬에서 실행 가능한가?
- [ ] **에러 없이 빌드되는가**: 처음 클론 후 `pnpm verify`가 통과하는가?

### B. 코드 탐색

- [ ] **폴더 구조가 직관적인가**: 파일을 찾기 위해 5초 이상 고민하지 않는가?
- [ ] **네이밍이 일관적인가**: 파일명, 함수명, 변수명이 예측 가능한가?
- [ ] **진입점이 명확한가**: 어디서부터 읽어야 하는지 알 수 있는가?
- [ ] **순환 의존이 없는가**: 모듈 간 의존 방향이 단방향인가?

### C. 개발 도구

- [ ] **린트/포맷이 자동인가**: 저장 시 또는 커밋 시 자동으로 코드 스타일이 통일되는가?
- [ ] **타입 체크가 즉각적인가**: IDE에서 빨간줄이 바로 뜨는가?
- [ ] **테스트 실행이 빠른가**: `pnpm test`가 10초 이내에 끝나는가?
- [ ] **핫 리로드가 동작하는가**: 코드 수정 후 브라우저가 자동 갱신되는가?

### D. 에러 메시지

- [ ] **에러 메시지가 행동 가능한가**: "뭐가 잘못됐고, 어떻게 고치라"가 있는가?
- [ ] **스택 트레이스가 의미 있는가**: 에러 발생 위치를 바로 찾을 수 있는가?
- [ ] **환경 문제 에러가 친절한가**: Node 버전, 패키지 누락 등을 명확히 안내하는가?

### E. 문서화

- [ ] **왜(Why)가 적혀 있는가**: 코드에 없는 맥락(결정 이유, 제약)이 문서에 있는가?
- [ ] **문서가 코드와 동기화되어 있는가**: 문서에 적힌 명령어가 실제로 동작하는가?
- [ ] **불필요한 문서가 없는가**: 읽을 필요 없는 문서가 혼란을 주지 않는가?

## 출력 포맷

```markdown
### Developer Experience 판단

| 항목 | 판정 | 근거 |
|---|---|---|
| 온보딩 | ✅/🟡/🔴 | ... |
| 코드 탐색 | ✅/🟡/🔴 | ... |
| 개발 도구 | ✅/🟡/🔴 | ... |
| 에러 메시지 | ✅/🟡/🔴 | ... |
| 문서화 | ✅/🟡/🔴 | ... |

### 종합: 🟢 진행 / 🟡 재검토 / 🔴 멈추고 논의
> {한 문장 판정 근거}
```

## 심각도 기준

| 심각도 | 기준 | 예시 |
|---|---|---|
| Critical | 셋업 불가, 빌드 실패, 문서 부재 | clone 후 실행 안 됨, README 없음 |
| Warning | 셋업에 수동 작업 필요, 문서 불일치 | .env 수동 복사 필요, 문서 명령어 오류 |
| Suggestion | DX 개선 여지 | 더 나은 에러 메시지, 폴더 정리 |
