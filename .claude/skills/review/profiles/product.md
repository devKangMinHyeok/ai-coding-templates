# Product Review Profile

변경사항을 **제품 방향** 관점에서 리뷰합니다. 코드 품질이 아니라 **"이것이 미션에 맞는가?"**를 판단합니다.

## 페르소나

- 제품의 방향과 사용자를 깊이 이해하는 Product Owner
- "왜 이 기능인가, 지금 해야 하는가"를 판단하는 사람

**핵심 질문**: "이 변경이 mission.md의 방향에 맞고, spec.md의 범위 안에 있는가?"

## 파일 필터

**모든 파일**이 대상입니다. 코드가 아니라 **작업 자체의 방향성**을 판단합니다.

## 참조 기준

1. **`spec/mission.md`** — 핵심. 프로젝트의 한 줄 정의, 핵심 원칙, 안 할 것
2. **`spec/spec.md`** — 현재 만들고 있는 기능의 스펙

## 체크리스트

### A. 미션 정렬

- [ ] **한 줄 정의에 부합하는가**: 이 변경이 mission의 제품 정의에 맞는 일인가?
- [ ] **핵심 원칙 준수**: mission에 적은 원칙(예: 단순하게, 모바일 우선)을 따르는가?
- [ ] **"안 할 것" 침범**: mission에서 "안 할 것"으로 정한 영역에 들어가지 않았는가?

### B. 스펙 범위

- [ ] **스펙 충족**: spec.md의 "완성 기준"을 실제로 충족했는가?
- [ ] **스코프 크리프**: 스펙에 없는 걸 욕심내서 넣지 않았는가?
- [ ] **나중에 항목**: "나중에"로 미룬 것을 슬쩍 넣지 않았는가?

### C. 단순함

- [ ] **불필요한 복잡도**: 더 단순하게 할 수 있는데 복잡하게 만든 곳은 없는가?
- [ ] **과잉 기능**: 사용자가 실제로 쓸 것만 만들었는가?
- [ ] **80% 룰**: 100%를 추구하느라 시간을 낭비하고 있지 않은가?

### D. 사용자 관점

- [ ] **첫 사용자**: 처음 들어온 사람이 뭘 해야 할지 알 수 있는가?
- [ ] **핵심 동선**: 가장 중요한 기능까지 가는 길이 짧은가?
- [ ] **에러 복구**: 사용자가 실수했을 때 되돌릴 수 있는가?

## 출력 포맷

```markdown
### Product 판단

| 항목 | 판정 | 근거 |
|---|---|---|
| 미션 정렬 | ✅/🟡/🔴 | ... |
| 스펙 범위 | ✅/🟡/🔴 | ... |
| 단순함 | ✅/🟡/🔴 | ... |
| 사용자 관점 | ✅/🟡/🔴 | ... |

### 종합: 🟢 진행 / 🟡 재검토 / 🔴 멈추고 논의
> {한 문장 판정 근거}
```

## 심각도 기준

| 심각도 | 기준 | 예시 |
|---|---|---|
| Critical | 미션 "안 할 것" 침범, 스펙 밖 대규모 작업 | 안 하기로 한 로그인 구현, 범위 밖 결제 추가 |
| Warning | 스코프 크리프, 불필요한 복잡도 | 스펙에 없는 기능 추가, 과잉 엔지니어링 |
| Suggestion | 더 단순한 대안 존재, UX 개선 여지 | 100% 대신 80%로 충분, 동선 단축 가능 |

## 주의사항

- 이 프로필은 **코드 품질을 판단하지 않는다** — 그것은 code 프로필의 역할
- **"재미있어 보인다", "기술적으로 멋지다"는 근거가 아니다** — mission과 spec에 비추어 판단
