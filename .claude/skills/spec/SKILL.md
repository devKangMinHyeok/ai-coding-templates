# /spec — 기능을 질문으로 좁힌다 (구현하지 않음)

## 역할
AI는 막연한 요청을 받으면 헤맨다. 이 스킬은 사용자를 **대화형으로 인터뷰**해서, 만들 기능을 `spec/spec.md` 한 장으로 좁힌다.
**여기서는 앱 코드를 절대 쓰지 않는다.** 스펙이 확정되면 `/impl`로 넘긴다.

## 전제
`spec/mission.md`가 있어야 한다. 없으면 **먼저 `/interview`부터** 하라고 안내하고 멈춘다.

## 절차 (질문은 한 번에 하나씩)

### 1. 무엇을 만들 건가
"어떤 기능을 만들고 싶나요? 한두 문장으로 말해주세요."

### 2. 핵심 기능
"꼭 되어야 하는 것을 우선순위로 적어주세요. (최대 3개)"

### 3. 완성 기준
"'여기까지 되면 성공'을 한 줄로 말한다면?"

### 4. 복잡도 체크
"다음 중 필요한 게 있나요?"
- 데이터 저장 (새로고침해도 남아야 함)
- 로그인
- 결제
- 외부 서비스 연동

→ 답이 모호하면 **그 지점만 더 깊게 되묻는다**(소크라테스식).

### 5. mission 정렬 확인
`spec/mission.md`를 읽고:
- "안 할 것"과 충돌하는 요구가 있으면 사용자에게 알린다.
- 형태(보여주기/기록관리/도구/갤러리)와 일치하는지 확인한다.

### 6. review profile 참조
스펙의 성격에 따라 관련 리뷰 프로필의 체크리스트를 미리 참고한다:
- 사용자 대면 기능 → `profiles/customer-support.md`의 에러 처리·안내 기준
- 마케팅 페이지 → `profiles/marketer.md`의 전환·카피 기준
- 서버/DB 관련 → `profiles/infra.md`의 배포·비용 기준

스펙에 이런 관점이 빠져 있으면 사용자에게 자연스럽게 되묻는다:
> "이 기능에서 에러가 나면 사용자에게 어떻게 안내할 건가요?"
> "이 데이터를 저장하면 무료 플랜 한도 내에서 괜찮을까요?"

### 7. 범위 좁히기
무거운 요구(로그인·결제·외부연동)는 빼서 `나중에` 항목으로 미룬다. 반드시 사용자와 합의.

## 마무리
- `spec/TEMPLATE.md` 형식으로 `spec/spec.md`를 쓴다(이미 있으면 누적/교체).
- 사용자에게 보여주고 "이대로 만들까요?" 확인.
- 확인되면: **"이제 `/impl`로 구현을 시작하세요"** 라고 안내하고 끝낸다. (직접 구현 금지)

## 자동 기록 (조용히, 확인 안 묻고 기록)

### decisions 자동 기록
스펙이 확정되면 다음을 `decisions/` 폴더에 자동 기록한다:
- **기능 범위 결정** → 무엇을 포함하고 무엇을 뺐는지, 그 이유
- **"나중에" 항목** → 무엇을 왜 미뤘는지 (로그인, 결제, 외부연동 등)
- **복잡도 선택** → DB 필요 여부, 외부 서비스 연동 여부와 이유

파일명: `decisions/YYYY-MM-DD-spec-{기능명}.md`, frontmatter에 `source: auto/spec`, `scope: feature`, `status: active`
