# /learn — 프로젝트 배움 기록

작업하면서 알게 된 것을 기록해두면, 다음 세션에서도 같은 삽질을 반복하지 않는다.

## 사용법

```
/learn              → 지금까지 기록된 배움 목록 (= /learn list)
/learn add          → 새로운 배움을 직접 기록한다 (인터뷰 형식)
/learn list         → 지금까지 기록된 배움 목록
/learn search <키워드>  → 키워드로 관련 배움을 찾는다
```

## 자동 기록 (다른 스킬에서 자동으로 쌓임)

**아래 상황에서 사용자가 `/learn add`를 하지 않아도 자동으로 기록된다:**

- **/impl 중 에러를 고쳤을 때** → 원인과 해결 방법을 `bug` 카테고리로 기록
- **/impl 중 새 패턴을 적용했을 때** → 어떤 패턴을 왜 썼는지 `pattern`으로 기록
- **/review에서 Critical/Warning을 발견했을 때** → 무엇이 잘못됐고 어떻게 고쳐야 하는지 기록
- **/verify 실패 후 수정했을 때** → 실패 원인과 수정 방법을 기록
- **/ship 완료 후** → 이번 작업의 주요 발견을 요약 기록

자동 기록 시 사용자에게 별도 확인을 묻지 않는다. 조용히 기록하고 넘어간다.

## `/learn add` 절차 (수동)

사용자에게 차례로 질문한다:

### 1단계: 무엇을 배웠나?
> "방금 뭘 배웠나요? 한 줄로 요약해주세요."

### 2단계: 어떤 상황이었나?
> "어떤 상황에서 이걸 알게 됐나요? (에러, 삽질, 코드 리뷰 등)"

### 3단계: 다음에 어떻게 적용할까?
> "다음에 비슷한 상황이 오면 어떻게 해야 할까요?"

### 4단계: 카테고리
> "어떤 종류인가요?"
> - `bug` — 버그·에러 해결
> - `pattern` — 코드 패턴·구조
> - `tool` — 도구·라이브러리 사용법
> - `decision` — 설계 결정 근거
> - `tip` — 기타 팁

### 5단계: 기록

파일명: `learnings/YYYY-MM-DD-{짧은-슬러그}.md`

## 기록 형식

```markdown
---
date: 2024-01-15
category: bug
source: auto/impl
---

# Supabase 연결 시 타임아웃 에러

## 상황
로컬에서 Supabase 연결 시 5초 이상 걸리면 타임아웃 에러 발생.

## 배움
`supabaseClient` 생성 시 `db.schema` 옵션이 누락되면 기본 스키마를 찾느라 느려짐.

## 다음에는
Supabase 클라이언트 생성 시 항상 `db: { schema: 'public' }` 명시.
```

- `source: auto/<스킬명>` — 자동 기록 시 어느 스킬에서 기록됐는지
- `source: manual` — `/learn add`로 직접 기록한 경우

## `/learn list` 절차

1. `learnings/` 폴더의 모든 `.md` 파일을 읽는다
2. 날짜 역순으로 정리한다
3. 테이블로 출력한다:

```markdown
| 날짜 | 카테고리 | 출처 | 제목 |
|---|---|---|---|
| 2024-01-15 | bug | auto/impl | Supabase 연결 시 타임아웃 에러 |
| 2024-01-14 | pattern | manual | 서버 액션에서 에러 처리 패턴 |
```

## `/learn search <키워드>` 절차

1. `learnings/` 폴더에서 키워드가 포함된 파일을 찾는다
2. 관련도 순으로 정리해서 출력한다

## 세션 시작 시 자동 참조

새 세션이 시작되면 `learnings/` 폴더가 있는 경우 최근 5개를 훑어보고, 현재 작업과 관련 있는 배움이 있으면 자연스럽게 참고한다. (사용자에게 목록을 덤프하지는 않는다)

## 폴더 구조

```
learnings/
├── 2024-01-15-supabase-timeout.md
├── 2024-01-14-server-action-error.md
└── ...
```

`learnings/` 폴더가 없으면 첫 기록 시 자동 생성한다.
