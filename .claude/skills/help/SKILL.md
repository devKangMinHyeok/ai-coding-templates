# /help — 도움이 필요할 때

사용자의 **현재 상태**를 파악해서 맞춤형으로 안내한다. 매뉴얼을 덤프하지 않는다.

## 사용법

```
/help              → 현재 상태를 보고 다음에 할 일을 안내
/help 커맨드       → 커맨드 목록과 간단한 설명
/help review       → /review 사용법 상세
/help 에러         → 에러 해결 도움
/help <아무 질문>  → 관련 안내
```

## 실행 흐름

### 1. 현재 상태 파악

다음을 조용히 확인한다 (결과를 나열하지 않는다):

```bash
# 미션이 있는가?
test -f spec/mission.md

# 스펙이 있는가?
test -f spec/spec.md

# 변경사항이 있는가?
git diff --name-only HEAD

# 검증이 통과하는가?
pnpm verify 2>/dev/null

# learnings/decisions가 있는가?
ls learnings/ decisions/ 2>/dev/null
```

### 2. 상태별 맞춤 안내

#### 미션 없음 (아직 시작 안 함)
> "아직 프로젝트 방향이 정해지지 않았어요. `/init`으로 시작하면 환경 정리 + 방향 잡기까지 한 번에 할 수 있어요. 이미 만들고 싶은 게 있으면 `/interview`로 바로 시작해도 됩니다."

#### 미션만 있음 (스펙 없음)
> "프로젝트 방향은 잡혀 있어요! 이제 구체적으로 어떤 기능을 만들지 정할 차례예요. `/spec`로 기능을 구체화해보세요."

#### 미션 + 스펙 있음 (구현 전)
> "스펙까지 준비됐어요. `/impl`로 구현을 시작하면 됩니다. 작게 나눠서 단계별로 만들어갈게요."

#### 구현 중 (변경사항 있음)
> "작업 중이시네요. 검증은 `/verify`, 막히면 뭐가 막혔는지 말해주세요."

#### 구현 완료 (검증 통과)
> "검증이 통과했어요! `/review`로 점검한 후 `/ship`으로 배포하면 됩니다."

### 3. 항상 포함하는 정보

상태 안내 후에 아래를 **짧게** 덧붙인다:

> **핵심 5단계**: `/interview` → `/spec` → `/impl` → `/review` → `/ship`
>
> 더 궁금한 게 있으면 편하게 물어보세요. "이거 어떻게 해?", "에러가 나요", "뭐가 잘못된 거야?" 뭐든 좋습니다.

## 특정 주제 도움

### `/help 커맨드` 또는 `/help 명령어`

모든 사용 가능한 커맨드를 간단히 표로 보여준다:

```markdown
| 커맨드 | 한 줄 설명 |
|---|---|
| `/init` | 템플릿 초기화 + 환경 소개 + 인터뷰 시작 (처음 한 번) |
| `/interview` | 프로젝트 방향(미션) 잡기 |
| `/spec` | 만들 기능 구체화 (스펙) |
| `/impl` | 스펙대로 구현 (단계별 검증) |
| `/verify` | 타입·린트·테스트·빌드 검증 |
| `/review` | 전문가 관점 점검 (8종 프로필) |
| `/ship` | 커밋·푸시·배포 |
| `/learn` | 배운 것 보기·기록 (자동으로도 쌓임) |
| `/decision` | 설계 결정 보기·기록 (자동으로도 쌓임) |
| `/add-review-profile` | 커스텀 리뷰 프로필 만들기 |
| `/help` | 지금 보고 있는 이것 |
```

### `/help review`

리뷰 프로필 8종과 사용법을 안내한다:

```markdown
`/review` — 변경사항을 전문가 관점으로 점검합니다.

| 프로필 | 관점 | 언제 쓰면 좋은가 |
|---|---|---|
| code | 코드 품질·타입·보안 | 항상 (자동 선택됨) |
| design | UI 일관성·상태·반응형 | 화면을 만들었을 때 |
| product | 미션 정렬·스펙 범위 | 기능이 방향에 맞는지 확인할 때 |
| developer | DX·온보딩·문서화 | README나 구조를 바꿨을 때 |
| marketer | 가치 제안·전환·카피 | 랜딩 페이지를 만들었을 때 |
| customer-support | 에러 안내·사용자 지원 | 에러 처리를 추가했을 때 |
| infra | 배포·환경변수·성능 | 서버/DB 코드를 바꿨을 때 |
| qa | 엣지 케이스·회귀·테스트 | 기능이 완성됐을 때 |

사용법: `/review`, `/review code`, `/review all`
```

### `/help 에러` 또는 에러 메시지를 포함한 질문

에러 메시지를 분석해서 해결 방법을 안내한다. 흔한 에러:

- **"pnpm: command not found"** → `npm install -g pnpm` 안내
- **"타입 에러" / 커밋 차단** → 에러 위치를 찾아서 수정 방법 안내
- **"모듈을 찾을 수 없습니다"** → `pnpm install` 안내
- **"NEXT_PUBLIC_SUPABASE_URL"** → `.env.local` 설정 안내

### 그 외 자유 질문

사용자가 "이거 어떻게 해?" 같은 자유 질문을 하면, 가장 관련 있는 스킬이나 가이드로 연결한다. 복잡한 설명 대신 "이렇게 하면 됩니다" + 1~2줄 예시.

## 톤

- **친절하되 간결하게.** 장문을 피하고, 핵심만.
- **"모르겠어도 괜찮아요"** 느낌. 비개발자가 부담 없이 물어볼 수 있게.
- **매뉴얼이 아니라 대화.** "다음은 이것입니다" 말고 "이거 해보시겠어요?"
- 사용자가 구체적으로 물어볼수록 구체적으로 답한다. 뭉뚱그려 물으면 큰 그림만.
