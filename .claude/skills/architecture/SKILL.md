# /architecture — 잘 자라는 구조로 짓는다

이 스킬은 **오래 가는 제품**을 만들기 위한 구조·의사결정 가이드다. 수업 당일뿐 아니라 이후 혼자 키울 때도 이 원칙을 따른다.

## 1. 의사결정 프레임 (모든 설계 판단의 첫 질문)

어떤 결정이든 먼저 둘로 나눈다.

- **결정적(기계가 채점 가능)** — 빌드 되나, 타입 맞나, 테스트 통과하나, 린트 지키나.
  → **자동 검증으로 강제**한다. 사람 판단에 맡기지 않는다. (`/verify`, `.husky/pre-commit`가 이 역할)
- **비결정적(판단이 필요)** — 어떤 구조로 갈지, 데이터를 어떻게 둘지, 무엇을 먼저 만들지.
  → **결정 전에 탐색 범위를 좁힌다.** 선택지를 2~3개로 줄이고, **가장 단순하고 되돌리기 쉬운 것**을 고른다.

> 황금률: **되돌리기 쉬운 결정은 빠르게, 되돌리기 어려운 결정은 늦게.**
> 비개발자일수록 "지금 가장 단순한 것"을 고르고, 정말 필요해질 때 키운다.

## 2. 코드는 어디에 두나 (폴더 구조)

```
app/            # 화면(라우트). 되도록 얇게 — 데이터 가져와 컴포넌트에 넘기는 정도
  page.tsx
  <기능>/page.tsx
  api/<이름>/route.ts   # 서버가 필요할 때만 (저장·외부연동·비밀키)
components/     # 재사용 UI 조각 (버튼, 카드, 목록 아이템…)
lib/            # 순수 로직·계산·데이터 함수 (화면과 분리)
```

**배치 원칙: 코드는 자신이 다루는 관심사를 소유한 곳에 둔다.**
- 계산·변환·규칙 → `lib/` (화면 없는 순수 함수)
- 보이는 것 → `components/`
- 라우트(`app/`)는 조립만. 로직을 라우트에 욱여넣지 않는다.

**의존 방향(한 방향으로만):** `app/`(화면) → `components/` → `lib/`(로직).
거꾸로(로직이 화면을 import) 가지 않는다. 그래야 안 엉킨다.

## 3. 자주 하는 결정 — 레시피

**데이터를 저장해야 하나?**
1. 새로고침해도 남기만 하면 됨 → **브라우저 로컬 저장(localStorage)**. (가장 단순, 서버 불필요)
2. 여러 기기/여러 사람이 공유해야 함 → 그때 DB 도입(예: Vercel과 연동되는 DB). 처음부터 하지 않는다.

**서버가 필요한가? (클라이언트 vs 서버)**
- 비밀키를 써야 하거나(외부 API 키), 남의 눈에 로직을 숨겨야 하면 → `app/api/.../route.ts`(서버).
- 그 외 화면 동작 대부분 → 클라이언트 컴포넌트로 충분.

**컴포넌트를 쪼갤 때**
- 한 파일이 화면에서 "두 가지 이상의 일"을 하면 분리.
- 같은 UI가 두 번 반복되면 `components/`로 추출.
- 너무 일찍 쪼개지 않는다 — 두 번째 등장할 때 추출(성급한 추상화 금지).

**새 기능을 추가할 때**
- 먼저 `/interview`로 스펙을 좁힌다 → `lib/`에 로직, `components/`에 UI, `app/`에 라우트.
- 한 번에 하나씩, 매번 `/verify`.

## 4. 같은 문제 두 번 겪지 않기
- 같은 버그·실수가 반복되면, 다음에 또 안 나도록 **구조로 박는다**: 검증에 항목 추가, 또는 규칙을 `conventions`에 한 줄 추가.
- "조심하자"는 다짐(비결정적)을 "검사로 막기"(결정적)로 옮기는 것이 핵심.

## 함께 보기
`conventions`(코드 규칙) · `ui-design`(화면 품질) · `verify`(검증) · `review`(배포 전 점검)

## 서버 코드라면
저장·외부연동·비밀키가 필요한 서버 코드는 **`server-architecture` 스킬**을 따른다. (도메인별 레이어 + 포트&어댑터)
