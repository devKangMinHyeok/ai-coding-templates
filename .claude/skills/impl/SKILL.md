# /impl — 스펙을 작게 나눠 구현한다

## 전제
- `spec/mission.md`가 있어야 한다. 없으면 **먼저 `/interview`부터**.
- `spec/spec.md`가 있어야 한다. 없거나 비어 있으면 **먼저 `/spec`부터** 하라고 안내하고 멈춘다.
- `decisions/` 폴더가 있으면 활성 결정을 훑어보고 관련 기술 결정을 참고한다.
- `learnings/` 폴더가 있으면 최근 배움 중 관련 있는 것을 참고한다.

## 절차
1. `spec/mission.md`와 `spec/spec.md`를 읽는다.
2. "꼭 되어야 하는 기능"을 **작은 단계로 쪼갠다**(한 단계 = 작고 검증 가능한 변경).
3. 각 단계마다:
   - 최소한으로 구현한다(많이 만들지 않는다).
   - **mission의 "안 할 것"과 충돌하면 멈추고 사용자에게 알린다.**
   - **`/verify`(= `pnpm verify`)를 돌려 초록불이 될 때까지 고친다.**
   - 초록불이면 다음 단계로.
4. 완성 기준(spec의 "오늘의 성공")을 충족하면 멈추고 보고한다.
5. 사용자에게 **"`/review`로 점검 후 `/ship`으로 배포하세요"** 안내.

## 자동 기록 (조용히, 확인 안 묻고 기록)

### learnings 자동 기록
다음 상황이 발생하면 `learnings/` 폴더에 자동으로 기록한다:
- **에러를 디버깅해서 고쳤을 때** → 원인과 해결 방법을 `bug` 카테고리로
- **새로운 패턴/구조를 적용했을 때** → 어떤 패턴을 왜 썼는지 `pattern`으로
- **/verify 실패 후 수정했을 때** → 실패 원인과 수정 방법을 `bug`로

파일명: `learnings/YYYY-MM-DD-{짧은-슬러그}.md`, frontmatter에 `source: auto/impl`

### decisions 자동 기록
다음 상황이 발생하면 `decisions/` 폴더에 자동으로 기록한다:
- **라이브러리를 새로 추가했을 때** → 왜 이 라이브러리인지, 대안은 뭐였는지
- **데이터 모델/구조를 설계했을 때** → 왜 이 구조인지
- **"나중에"로 미룬 것이 있을 때** → 무엇을 왜 미뤘는지
- **구현 방식을 선택했을 때** (서버 vs 클라이언트, 캐시 전략 등)

파일명: `decisions/YYYY-MM-DD-{짧은-슬러그}.md`, frontmatter에 `source: auto/impl`, `status: active`

### 기록하지 않는 것
- 사소한 타이포 수정, import 정리 등 판단이 필요 없는 것
- 이미 기록된 것과 중복되는 것

## 원칙
- **작게, 자주 검증.** 큰 덩어리로 만들고 끝에 한 번 검증하지 않는다.
- 검증이 초록불이 되기 전에는 그 단계를 "끝났다"고 하지 않는다.
- spec 범위 밖의 욕심(스코프 크리프)은 `나중에`로 미루고 사용자에게 묻는다.
- **mission의 "안 할 것"은 절대 넘지 않는다.** 사용자가 요청해도 먼저 mission 수정을 권한다.
